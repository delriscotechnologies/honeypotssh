SSH Honeypot Lab with Cowrie and Hydra

Introduction

In a landscape where attackers continuously evolve and probe networks for misconfigurations and weak credentials, defenders must stay one step ahead—not just with tools, but with mindset. This project was born from a simple premise: if we can simulate an attacker in a controlled environment, we can learn to detect and respond to them more effectively in the real world.

This lab features a two-machine setup: one Linux Mint VM running Cowrie, a powerful SSH honeypot that simulates a real shell environment; and a Kali Linux VM using Hydra to simulate brute-force login attempts. By combining these perspectives, we create a safe environment to study attacker behavior and log fidelity.

Cowrie logs every detail—connection attempts, used credentials, commands issued, and session flow—making it an ideal tool for studying threat activity without risking production systems. Meanwhile, Hydra reveals just how quickly and efficiently weak login combinations can be exploited when protections are absent or misconfigured.

But this wasn’t just a copy-paste deployment. I configured, broke, fixed, and fine-tuned every piece. This lab helped me sharpen both my offensive intuition and defensive insight, and reminded me that the best way to understand security is to build and observe it firsthand.

Linux Mint Honeypot Configuration

The Linux Mint machine (192.168.7.199) is prepared to run Cowrie with the following steps:

sudo apt update && sudo apt upgrade -y
sudo apt install python3-pip python3-venv libssl-dev libffi-dev build-essential libpython3-dev authbind -y

Run Cowrie from a dedicated, non-root account:

sudo adduser --disabled-password cowrie
sudo -iu cowrie

Create a self-contained state directory and activate its virtual environment:

mkdir ~/my-honeypot && cd ~/my-honeypot
python3 -m venv cowrie-env
source cowrie-env/bin/activate

Install and initialize the reviewed Cowrie release:

python -m pip install --upgrade pip
python -m pip install "cowrie==3.0.0"
cowrie init
nano etc/cowrie.cfg

Safety boundary: keep the honeypot on an isolated lab network, run it as the dedicated non-root account shown above, and never store real credentials in etc/userdb.txt.

Exit the Cowrie account, then redirect incoming port 22 traffic to Cowrie’s port 2222 from an administrator account. Move the real SSH service to another port first so you do not lock yourself out:

exit
sudo iptables -t nat -A PREROUTING -p tcp --dport 22 -j REDIRECT --to-port 2222
sudo apt install iptables-persistent -y
sudo netfilter-persistent save

Return to the Cowrie account, activate the environment, and start the honeypot:

sudo -iu cowrie
cd ~/my-honeypot
source cowrie-env/bin/activate
cowrie start
cowrie status

Monitor live logs:

tail -f var/log/cowrie/cowrie.log

Kali Linux Attack Simulation with Hydra

Hydra is used on Kali to perform brute-force attacks on Cowrie:

hydra -h

List common wordlists:

ls -al /usr/share/wordlists/

Attack using a known username:

hydra -l admin -P /usr/share/wordlists/rockyou.txt -t 4 -f ssh://192.168.7.199

Attack using unknown usernames:

hydra -L /usr/share/wordlists/rockyou.txt -P /usr/share/wordlists/top-passwords-shortlist.txt -t 4 -f ssh://192.168.7.199

Cowrie logs all attempts, giving full visibility into attacker methods and brute-force timing. These logs include IP, user/pass combinations, and interactive commands entered by the attacker.

Final Thoughts & Reflections

This exercise brought to life the duality of red and blue team thinking. As I watched Hydra hammer Cowrie, I realized how quickly bad actors can find a way in—if given the opportunity. But more importantly, I saw how honeypots can serve as an early warning system, a data goldmine, and a training ground for detection strategies.

One helpful configuration tip: Cowrie accepts fake credentials listed in:

etc/userdb.txt

This defines "valid" credentials attackers can use to gain access. Tuning this file helps simulate realistic login behavior that makes the honeypot appear more convincing.

Personally, this project gave me more than just logs—it gave me perspective. It showed me how security is both about defense and observation. I now see SSH not just as a secure protocol, but as an attack surface rich with signals. I learned how to read those signals, how to record them, and how to think critically about what they mean.

What I built was simple. But what I learned was powerful: every alert starts with a packet. Every log tells a story. And every story teaches you something—if you’re paying attention.
